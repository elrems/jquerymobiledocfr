Le plugin fixedtoolbar a les options suivantes :

visibleOnPageShow boolean

Par défaut : true

Cela détermine si la barre d'outils est visible ou non lorsque sa page parent est affichée. Cette option est également définie comme un attribut de données : data-visible-on-page-show="false"

$("[data-role=header]").fixedtoolbar({ visibleOnPageShow: false });
disablePageZoom boolean

Par défaut : true

Cela détermine si le redimensionnement utilisateur doit être désactivée sur les pages qui contiennent des barres d'outils fixes. Cette option est également définie comme un attribut de données : data-disable-page-zoom="false"

$("[data-role=header]").fixedtoolbar({ disablePageZoom: false });
transition string

Par défaut : "slide" (ce qui finit par utiliser slideup et slidedown)

La transition qui devrait être utilisé pour afficher et masquer une barre d'outils fixes. Les valeurs possibles sont "none", "fade" et "slide" (ou vous pouvez écrire votre propre transition CSS et l'utiliser). Cette option est également définie comme un attribut de données : data-transition=""fade""

$("[data-role=header]").fixedtoolbar({ transition: "fade" });
fullscreen boolean

par défaut : false

Les barres d'outils "plein écran" se positionnent en haut du contenu à chaque fois qu'elles sont visibles, et à la différence des barres d'outils fixes habituelles, les barres d'outils "plein écran" ne se replient pas vers un positionnement statique lors d'un basculement (afficher/masquer), au lieu de cela elles disparaissent entièrement de l'écran. Les barres d'outils "plein écran" sont idéales pour les interfaces plus immersive, comme une visionneuse de photos qui est censé remplir tout l'écran avec la photo. Cette page montre les barres d'outils qui utilise l'option fullscreen. Cette option est également définie comme un attribut de données : data-fullscreen=""true""

$("[data-role=header]").fixedtoolbar({ fullscreen: true });

Remarque : Bien que la syntaxe de l'attribut de donnée pour cette option n'a pas changé, il est désormais supporté uniquement sur l'élément de la barre d'outils, et non plus sur l'élément de la page.

tapToggle boolean

Par défaut : true

Active ou désactive la capacité de l'utilisateur pour basculer la visibilité de la barre d'outils avec une tape sur l'écran (ou un clic, pour les utilisateurs de souris). Cette option est également définie comme un attribut de données : data-tap-toggle=""true""

$("[data-role=header]").fixedtoolbar({ tapToggle: true });

Remarque : Ce comportement était autrefois configurable comme suit, mais depuis la version 1.1, cette syntaxe n'est plus pris en charge : 

	
$.mobile.fixedToolbars
   .setTouchToggleEnabled(false);
tapToggleBlacklist string

Par défaut : "a, button, input, select, textarea, .ui-header-fixed, .ui-footer-fixed"

Une liste de sélecteurs jQuery qui, lorsqu'on les frappe, n'entraînera pas le basculement des barres d'outils.

$("[data-role=header]").fixedtoolbar({ tapToggleBlacklist: "a, button, input, select, textarea, .ui-header-fixed, .ui-footer-fixed" });
hideDuringFocus string

Par défaut : "input, select, textarea"

Une liste des sélecteurs jQuery qui devrait entraîner les barres d'outils à se masquer à l'exception des barres d'outils fixes.

$("[data-role=header]").fixedtoolbar({ hideDuringFocus: "input, select, textarea" });
updatePagePadding boolean

Par défaut : true

Comme les barres d'outils peuvent varier en hauteur en fonction du contenu qu'elles contiennent, cette option met à jour automatiquement l' espacement sur l'élément page pour s'assurer que les barres d'outils fixes ont suffisamment d'espace dans le document quand elles sont positionnées de façon statique, et lorsqu'on défile vers le haut ou le bas de la page. Lorsque cela est activé, l'espacement se met à jour durant plusieurs opérations, telles que pageshow, lors des transitions de page et sur resize et orientationchange. Comme optimisation, nous recommandons de désactiver cette option et d'ajouter une règle à votre CSS pour l'espacement du div de la page pour correspondre à la hauteur EM de vos barres d'outils, comme ceci .ui-page-header-fixed { padding-top: 4.5em; }. Cette option est également définie comme un attribut de données : data-update-page-padding="false"

$("[data-role=header]").fixedtoolbar({ updatePagePadding: false });
supportBlacklist function

Par défaut : une fonction qui retourne une valeur booléenne

Le support du CSS position: fixed est très difficile à tester. En fait, au moment de la version 1.1, il n'y avait aucun moyen connu de façon raisonnable pour tester le support de fixed dans certains navigateurs populaires. Cette option est une fonction qui tente de se retirer sur certaines plateformes populaires qui sont connues pour être gênantes avec position: fixed . Souvent, ces plateformes supportent partiellement position: fixed, ce qui peut être pire que de ne pas le supporter du tout. Si cette option primordiale fait partie de votre propre liste noire, il vous suffit de fournir une fonction qui renvoie un résultat vrai ou faux lorsqu'elle est appelée lors de l'initialisation. Vous devez la mettre dans mobileinit, de sorte qu'elle s'applique lorsque le plugin est initialement créé.


$( document ).on("mobileinit", function(){
  $.mobile.fixedtoolbar.prototype.options.supportBlacklist = function(){
    var result;
    // logique pour déterminer si le résultat doit être vrai ou faux
    return result;
  };
})
initSelector CSS selector string

Par défaut : ":jqmData(position='fixed')"

Ceci est utilisé pour définir les sélecteurs (types d'éléments, les data roles, etc) qui seront automatiquement initialisés comme des barres d'outils fixes. Pour changer les éléments qui sont initialisés, liez cette option à l'événement mobileinit :

$( document ).on( "mobileinit", function(){
	$.mobile.fixedtoolbar.prototype.options.initSelector = ".myselector";
});
