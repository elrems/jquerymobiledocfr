Dans les navigateurs qui supportent position: fixed, (la plupart des navigateurs de bureaux, iOS5+, Android 2.2+, BlackBerry 6, et d'autres), les barres d'outils qui utilisent le plugin "fixedtoolbar" seront fixées à la partie supérieure ou inférieure de la fenêtre, tandis que le contenu de la page défilera librement entre les deux. Dans les navigateurs qui ne supportent pas le positionnement fixe, les barres d'outils resteront positionnées dans le flux, en haut ou en bas de la page.

Pour activer ce comportement sur ​​une entête ou un pied de page, ajoutez l'attribut data-position="fixed" à un élément header ou footer de JQuery Mobile.

Exemple de balisage d'une entête fixe :

	
<div data-role="header" data-position="fixed">
	<h1>Entête fixe !</h1>
</div>

Exemple de balisage d'un pied de page fixe :

	
<div data-role="footer" data-position="fixed">
	<h1>Pied de page fixe !</h1>
</div>

Barres d'outils pleines écrans

Les barres d'outils plein écran se positionnent en haut du contenu à chaque fois qu'elles sont visibles, et à la différence des barres d'outils fixes habituelles, les barres d'outils "plein écran" ne se replient pas vers un positionnement statique lors d'un basculement. Au lieu de cela, elles disparaissent entièrement de l'écran. Les barres d'outils "plein écran" sont idéales pour les interfaces plus immersive, comme une visionneuse de photos qui est censé remplir tout l'écran avec la photo.

Pour activer cette option sur une entête ou un pied de page fixe, ajouter l'attribut data-fullscreen à l'élément.

	
<div data-role="header" data-position="fixed" data-fullscreen="true">
	<h1>Entête fixe !</h1>
</div>

Formulaires dans les barres d'outils

Bien que tous les éléments de formulaire sont maintenant testés pour fonctionner correctement dans les barres d'outils statiques pour jQuery Mobile 1.1, nous vous recommandons de nombreux tests lors de l'utilisation des éléments de formulaire dans les barres d'outils fixed ou dans tous les éléments position: fixed. Cela peut potentiellement déclencher un certain nombre de problèmes imprévisibles dans différents navigateurs mobiles, Android 2.2/2.3 en particulier (détaillé dans les Problèmes connus dans Android 2.2/2.3).

Les changements dans jQuery Mobile 1.1

Avant la version 1.1, jQuery Mobile utilisait dynamiquement les barres d'outils repositionnées pour donner l'effet d'une entête fixe, car très peu de navigateurs mobiles supportaient la propriété CSS position:fixed. Et en simulant une fixation grâce à l'utilisation de "faux" JavaScript, le comportement du débordement du défilement réduisait le nombre de navigateur supporté, en plus cela se comportait de façon anormale sur certaines plateformes. Ce comportement n'était pas idéal et jQuery Mobile 1.1 a pris une nouvelle approche pour les barres d'outils fixes qui permet un support beaucoup plus large. Le framework offre de véritables barres d'outils fixes sur les nombreuses plateformes populaires, au lieu d'un positionnement statique dégradé pour les plateformes ne pouvant le supporter.

Polyfill pour les anciennes plateformes

Le plugin de barre d'outils fixe se dégrade gracieusement dans les plateformes qui ne supportent pas correctement le CSS position:fixed , comme l'iOS4.3. Si vous avez encore besoin de supporter les barres d'outils fixes sur cette plateforme (avec le comportement afficher/Masquer) qui était incluse dans les versions précédentes, Filament Group a développé un polyfill que vous pouvez utiliser.

  • C'est le dépôt du code Github avec les CSS et JavaScript requis pour le polyfill des barres d'outils fixes
  • L'URL de démonstration en utilisant le code dans le dépôt ci-dessus

Il suffit d'inclure les fichiers CSS et JS après vos références à jQuery Mobile et les barres d'outils fixes travailleront de la même manière que jQuery Mobile 1.0 dans iOS4.3, avec l'ajout de la nouvelle API pour le plugin fixedtoolbar 1.1.

Si vous avez des améliorations à suggérer, faites un fork du gist sur github et faites le nous savoir !

Problème connu avec les contrôles de formulaire à l'intérieur des barres d'outils fixes et de scroll par programmation

Un problème obscur existe dans iOS5 et certaines plateformes Android où le contrôle de formulaire placé à l'intérieur de conteneurs fixes peut perdre sa zone active lorsque la fenêtre est par programmation scroller (en utilisant window.scrollTo par exemple). Ce n'est pas un problème spécifique à jQuery Mobile, mais à cause de cela, nous recommandons de ne pas faire défiler un document par programmation lors de l'utilisation des contrôles de formulaire à l'intérieur de barre d'outils fixe de jQuery Mobile. Ce ticket du projet des bugs d'appareils explique ce problème plus en détail.

Problèmes connus dans Android 2.2/2.3

La mise en œuvre de position: fixed; dans Android 2.2/2.3 peut, en conjonction avec des styles apparemment sans rapport avec des modèles de balisage, provoquer un certain nombre de problème étrange en particulier dans le cas des éléments position: absolute à l'intérieur d'éléments position: fixed. Bien que nous ayons fait de notre mieux pour contourner un certain nombre de ces bugs uniques relevant de la bibliothèque, notez que des styles personnalisés peuvent causer un certain nombre de problèmes.

  • Les éléments de formulaire situés ailleurs sur la page des menus de sélection en particulier, peuvent ne pas répondre à une interaction utilisateur quand un élément vide positionné en absolu est placé dans un élément à position fixe. Dans de rares cas et spécifiques à Android 2.2, cela peut causer sur la totalité des pages un échec à une interaction utilisateur. Cela peut apparemment être résolu en ajoutant n'importe quel caractère à l'élément positionné en absolu, y compris un espace insécable, et même dans certains cas des espaces.
  • Le problème ci-dessus peut également être déclenché par une image positionné en absolu à l'intérieur d'un élément en position fixe, mais seulement lorsque cette image est utilisée avec autre chose que ses dimensions inhérentes. Si une hauteur ou une largeur est spécifiée sur l'image en utilisant le CSS, ou si le src de l'image n'est pas valide (donc n'ayant pas de hauteur et de largeur inhérente), ce problème peut se produire. Si une image par exemple de 50x50 pixels est placée dans un élément fixe et est laissé à ses dimensions inhérentes, cette question ne semble pas se produire.
  • Quand un élément position: fixed apparaît n'importe où sur une page, la plupart des transformations 2D CSS échoueront. Curieusement, seuls les transformations translate semblent ne pas être affectées par cela. Encore plus étrange, ce problème est résolu en réglant le CSS opacity à .9 ou moins sur le parent de l'élément fixe.
  • La combinaison de position: fixed et des propriétés overflow est à éviter, car les deux ensembles sont connus pour causer des problèmes imprévisibles dans les anciennes versions de l'OS Android.
  • Tout élément qui déclenche le clavier à l'écran, lorsqu'il est placé dans un élément position: fixed, ne parviendra pas à répondre à la saisie de l'utilisateur lorsque vous utilisez autre chose que le clavier par défaut. Cela inclut Swype, XT9 ou, à priori, toute méthode de saisie en dehors de la norme non-intuitive du clavier.

Alors que nous allons continuer à essayer de trouver des façons d'atténuer ces bugs du mieux que nous le pouvons, nous vous conseillons actuellement lors de l'implémentation des barres d'outils fixes contenant des styles utilisateur compliqués et des éléments de formulaire de faire de nombreux essais dans toutes les versions du navigateur natif d'Android.

Les pages suivantes sont proposées pour tester les barres d'outils fixes et les éléments de formulaires : application démo, saisies de texte, recherche, radio bouton, case à cocher, curseur, sélection et boutons.

N'est plus pris en charge : touchOverflowEnabled

Avant la version 1.1 de jQuery Mobile, le véritable support de la barre d'outils fixe dépendait du support natif du navigateur avec la propriété CSS overflow-scrolling: touch, qui est actuellement pris en charge que par iOS5. Depuis la version 1.1, jQuery Mobile n'utilise plus du tout cette propriété CSS. Nous avons enlevé tous les usages internes de cette propriété dans le framework, toutefois, nous avons laissé l'objet $.mobile défini globalement pour éviter que sa suppression puisse causer des problèmes avec les applications existantes. Cette propriété est marquée pour suppression, donc s'il vous plaît mettez à jour votre code pour ne plus l'utiliser. Cependant, le test du support pour cette propriété reste défini sous $.support et nous n'avons pas l'intention d'enlever ce test pour l'instant.
